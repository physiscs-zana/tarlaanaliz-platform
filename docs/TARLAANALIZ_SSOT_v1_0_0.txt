## [KR-015] Drone Pilotları (Uçuş Operasyonları)

**Üyelik/Kayıt:** İl, ilçe, ad soyad, telefon; drone modeli ve seri numarası; hizmet verdiği mahalle/köy listesi (başlangıçta opsiyonel). Drone seri numarası doğrulama referansıdır.

**Günlük Veri Teslimi:** Uçuş çıktısı her gün hafıza kartı ile yerel istasyona aktarılır. **Her uçuşta farklı hafıza kartı kullanılır.**

> **Not:** DJI tarafındaki kart kilidi/şifre/PIN gibi özellikler cihaz ve sürüme göre değişebilir; güvenlik garantisi olarak değil, **sadece ek katman** olarak değerlendirilir.

**Her Aktarım Log Üretir:** pilot kimliği, drone seri no, tarih/saat, batch id, tarla eşleştirme durumu.

**Eşleştirme Zorunlu:** Yüke gelen veri seti mutlaka kayıtlı bir tarla ile otomatik eşleşir; minimum: il, ilçe, mahalle/köy, ada, parsel, bitki türü (varsa FieldID). **Eşleştirme yoksa işlenmez.**

## [KR-017] YZ Modeli ile Analiz (Kanonik: İzolasyon + Tek Yönlü Akış + Job Semantiği)

Bu KR, YZ analiz hattının **şemsiye** tanımıdır. Aşağıdaki KR’ler KR‑017’nin **operasyonel ayrıştırmasıdır**:
- **[KR-070]**: Worker izolasyonu (inbound kapalı) + egress allowlist (network policy)
- **[KR-071]**: One-way data flow + allowlist’in doğru yeri (Ingress) + mTLS cihaz kimliği
- **[KR-072]**: Dataset lifecycle + chain-of-custody (manifest/hash/signature/verification) — “hileyi ispat” katmanı
- **[KR-073]**: Untrusted file handling + AV1/AV2 + sandbox dönüşüm + quarantine

[ZORUNLU]
- AnalysisJob kimliği: **FieldID + CropType + MissionID (varsa)**; **PII taşınmaz**
- Bitkiye özel YZ modeli routing (yönlendirme) ile otomatik seçilir. Bu karar; **MissionID, FieldID ve bitki türü** ile kayıt altına alınır (`model_id`, `model_version`).
- Pilot tarafında görünen görev bilgisi: **MissionID, il/ilçe/mahalle-köy, ada/parsel (ParcelRef), bitki türü, hedef tarama penceresi, teslim son tarihi**. **Çiftçi adı/telefonu gösterilmez**
- Hafıza kartları yerel istasyonda güvenlik kontrollerinden (**[KR-072]**, **[KR-073]**) geçmeden analize girmez: hash manifest + AV tarama + karantina kararları + allowlist uzantılar.
- EdgeKiosk/istasyon tarafında **model çalıştırma yoktur** (model hırsızlığı riskine karşı).
- Worker (YZ/processing) **inbound istek kabul etmez** (**[KR-070]**); job yalnızca dispatch/queue ile alınır. Web (PWA) ve public backend Worker ağına **doğrudan erişemez**.
- Worker, **[KR-018 / KR-082]** kalibrasyon “hard gate” şartı sağlanmadan işi kabul etmez (`requires_calibrated=true`) ve dataset kanıt zinciri doğrulanmadan çalıştırmaz (**[KR-072]**).
- Sonuçlar **kesinlikle tek yönlü** yayımlanır (**[KR-071]**): **FieldID + CropType + katmanlar + skor/uyarı** → sonuç servisi → web (PWA).
- Erişim denemeleri/deny olayları ve kritik adımlar audit/WORM log’a düşer (bkz. **BÖLÜM 3**).

### KR-017 Kabul Testleri (E2E)
- Edge’de model çalıştırma denemesi → build/CI policy ile **yasak** (repo’da model ağırlıkları bulunamaz)
- Worker’a inbound HTTP denemesi → **başarısız** (policy deny)
- Calibrated olmayan dataset ile job → Worker **reddeder**
- Manifest hash mismatch → dataset **REJECTED_QUARANTINE**

## [KR-018 / KR-082] RADIOMETRY - Tam Radyometrik Kalibrasyon Zorunluluğu (Radiometric Calibration: ışık/sensör etkilerini düzeltme)

## Amaç
- Model eğitimi (training: modelin öğrenmesi) ve saha sonuçları arasında tutarlılık (training-serving parity: eğitim/çalıştırma aynı dağılım).
- Müşteri memnuniyeti için zaman serisi (time-series: haftalık takip) ve tarlalar arası kıyasın (cross-field comparison: farklı tarlaları karşılaştırma) güvenilir olması.

## Kural (Hard Gate: sert kapı)
- **Calibrated Dataset (kalibre veri seti)** üretilmeden **AnalysisJob (analiz işi)** başlatılamaz.
- Worker (AI/Processing: yapay zekâ işleyici) **ham DN (Digital Number: sensör ham sayısal değeri)** veya kalibrasyonu belirsiz veriyi kabul etmez.

## İstasyon Akışı (Real-world: saha şartlarına uygun)
1) **Offline Security PC (internetsiz güvenlik makinesi)**:
   - AV (Anti-Virus: zararlı yazılım taraması) + manifest/hash (bütünlük kanıtı) + quarantine (karantina: izolasyon)
2) **Online Producer Workstation (izole internetli kalibrasyon makinesi)**:
   - Producer (DJI Terra / Pix4Dfields vb.) ile **tam radyometrik kalibrasyon**
   - Reflectance Panel (yansıma paneli: ışığı bilinen hedef) kullanımı SOP’a (Standard Operating Procedure: standart iş akışı) göre zorunlu
   - Processing Report (işleme raporu) + Calibration Result (kalibrasyon çıktısı) + QC Report (Quality Control: kalite raporu)
3) **Dispatch/Upload (gönderim)**:
   - Calibrated Dataset + raporlar + manifest/hash → Worker’a gönder

## Platform Sorumluluğu (Backend: sunucu)
- Calibration Gate (kalibrasyon kapısı): calibrated yoksa job açma
- QC Gate (kalite kapısı): PASS/WARN/FAIL (geç/uyarı/kal) kararını sakla, UI’a taşı, audit trail (denetim izi) üret

## Çıktı Sözleşmeleri (Contracts: ortak veri şemaları)
- CalibrationJob / CalibrationResult
- QCReport (pass_warn_fail + flags + recommended_action)
- AnalysisJob yalnızca `requires_calibrated=true` ile açılır

## [KR-033] Ödeme ve Manuel Onay (Müşteri Tahsilat Akışı)

**Amaç:** Çiftçi / Kooperatif / Üretici Birliği tarafından açılan **tek seferlik Mission** veya **yıllık Subscription** taleplerinde tahsilatı standartlaştırmak ve talebin “işlenebilir” hale gelmesini **ödeme onayına** bağlamak.

> ⚠️ **PII Notu:** Kimlik modeli gereği kullanıcı e-postası sistemde zorunlu alan değildir. IBAN/EFT ile ödeme senaryosunda kullanıcı dekontu **odeme@tarlaanaliz.com** adresine kendi e-postasıyla gönderebilir; platform DB’de e-posta adresi tutulmaz. Dekont/ek yönetimi için platform DB’de yalnızca `receipt_blob_id` gibi opak bir referans tutulur (object store / PII vault).  

### 1) Ödeme Yöntemleri (Payment Method)

- `CREDIT_CARD` (Kredi kartı / online): Ödeme sağlayıcısı üzerinden tahsilat (hosted checkout / payment session).  
- `IBAN_TRANSFER` (Havale/EFT): Kullanıcı IBAN’a transfer yapar; açıklamaya `PaymentRef` yazar; dekontu e-posta ile iletir; **Merkez Yönetim (CENTRAL_ADMIN)** manuel onay verir.

### 2) Ödeme Durumları (Payment Status)

`PaymentIntent.status` kanonik durumları:

- `PAYMENT_PENDING` → ödeme bekleniyor (intent açıldı, tahsilat doğrulanmadı)
- `PAID` → ödeme tamamlandı (provider webhook veya manuel admin onayı)
- `REJECTED` → ödeme reddedildi (dekont/uyuşmazlık/eksik bilgi)
- `EXPIRED` → süre aşıldı (örn. 7 gün içinde ödeme gelmezse)
- `CANCELLED` → ödeme intent’i iptal edildi (kullanıcı/admin; **PAID olmadan**)
- `REFUNDED` → ödeme iade edildi (**yalnızca PAID sonrası**)

**Kural-1 (Mission):** `PAID` olmadan Mission **ASSIGNED** olamaz.  
**Kural-2 (Subscription):** `PAID` olmadan Subscription **ACTIVE** olamaz; `PENDING_PAYMENT` durumunda scheduler Mission üretmez.  
**Kural-3 (İptal):** `PAYMENT_PENDING` iken kullanıcı `CANCELLED` yapabilir.  
**Kural-4 (İade):** `REFUNDED` yalnızca `PAID` sonrası admin aksiyonu ile olur; iade edilen ödeme, hedefe göre aşağıdaki politikayı tetikler.

### 3) Müşteri Tahsilat Akışı (Payment Flow)

**Tek seferlik analiz (Mission)**
1. Kullanıcı Mission talebi oluşturur → sistem `price_snapshot_id` üretir.
2. Sistem bir `PaymentIntent` üretir (`PAYMENT_PENDING`) ve kullanıcıya yöntem seçtirir.
3. `CREDIT_CARD`: hosted checkout → webhook/redirect → `PAID` → Mission “işlenebilir” hale gelir.
4. `IBAN_TRANSFER`: uygulama IBAN + `PaymentRef` + dekont e-posta talimatı gösterir → admin manuel kontrol → `PAID` veya `REJECTED`.
5. `PAID` olunca Mission atama kuyruğuna girer (PLANNED/ASSIGNED akışına uygun).

**Yıllık abonelik (Subscription)**
1. Kullanıcı aboneliği başlatır → Subscription `PENDING_PAYMENT` açılır, `price_snapshot_id` yazılır.
2. Sistem Subscription hedefli bir `PaymentIntent` üretir (`PAYMENT_PENDING`).
3. `PAID` olunca Subscription `ACTIVE` olur ve scheduler çalışır.

**IBAN Talimat Ekranı (PWA)**
- IBAN (kopyala)
- Alıcı adı
- Tutar
- `PaymentRef` (kopyala)
- “Açıklamaya PaymentRef yaz; dekontu odeme@tarlaanaliz.com adresine gönder; konu satırı PaymentRef olsun.”

### 7) Bildirimler (SMS + Uygulama İçi)

- `PAID` → “Ödemeniz alındı. Talebiniz işleme alındı.”
- `REJECTED` → “Ödemeniz reddedildi. Neden: {reason}”
- `EXPIRED` → “Ödeme süresi doldu. Ref: {PaymentRef}”
- `CANCELLED` → “Talebiniz iptal edildi. Ref: {PaymentRef}”
- `REFUNDED` → “Ödemeniz iade edildi. Ref: {PaymentRef}”

### 8) İptal / İade Politikası (Kanonik)

- **İptal (CANCELLED):** yalnızca `PAYMENT_PENDING` iken yapılır.  
- **İade (REFUNDED):** yalnızca `PAID` iken admin aksiyonu ile yapılır.
- **İade + Mission ilişkisi:** Mission henüz `FLOWN` aşamasına gelmediyse Mission `CANCELLED` yapılabilir; `FLOWN/UPLOADED/ANALYZING/DONE` sonrası iade kararı ticari politikadır ama sistemde `REFUNDED` tutulabilir (geri ödeme ayrı gateway işlemi olabilir).
- **İade + Subscription ilişkisi:** Subscription `ACTIVE` iken iade edilirse Subscription `CANCELLED` yapılır ve yeni Mission üretilmez (mevcut Missionlar ayrıca iptal edilebilir).

- `PAYMENT.REJECTED`
- `PAYMENT.EXPIRED`
- `PAYMENT.CANCELLED`
- `PAYMENT.REFUNDED`

> Bu olaylar BÖLÜM 3 log standardına uygun şekilde JSONL olarak üretilir; `correlation_id` zorunludur; PII yoktur.

- SAST/Dependency scan: bilinen kritik zafiyet yok (CVSS yüksek olanlar blok)
- Container scan + SBOM: imaj taraması temiz; SBOM üretilir ve arşivlenir

### Release Gate (prod'a çıkış)
- Sürümleme: release notları + migration planı; breaking change varsa sürüm artışı ve geriye uyumluluk stratejisi
- Smoke test: uçtan uca örnek akış (Mission create → ingest → worker → sonuç → harita katmanı) başarılı
- Sertifika yönetimi: kiosk sertifikaları basılı/dağıtılmış; rotasyon takvimi kayıtlı
- Rollback planı: veri bütünlüğünü bozmadan geri dönüş adımları tanımlı

### Ops Checklist (günlük/haftalık)
- Karantina kuyruğu: bekleyen dosya/mission sayısı ve sebepleri raporlanır
- Audit/WORM log doğrulaması: logların değişmediği kontrol edilir, saklama süresi politikası izlenir
- Anahtar/sertifika rotasyonu: süre dolumu yaklaşanlar listelenir, planlı yenileme yapılır
- Anomali izleme: olağandışı ingest hızı, tekrar eden başarısız doğrulama, şüpheli IP/cihaz hareketleri alarm üretir

## [KR-070] YZ Analiz İzolasyonu (Worker Isolation & Egress Policy)

**Amaç:** YZ (Worker) altyapısının web/platform tarafından “sorgulanabilir bir servis” gibi kullanılmasını engellemek; model/ham veri yüzeyini küçültmek; sahada ve merkezde güvenliği “varsayılan reddet” (deny-by-default) ile sabitlemek.

### Zorunlu Kurallar (Hard)
1) **AI Worker Inbound kapalı:** AI Worker (YZ analiz servisi) hiçbir koşulda inbound HTTP kabul etmez. Job alma yalnızca **dispatch/queue + pull/poll** ile olur.
   **Ingest Gateway ayrımı:** Sunucu 2’deki Ingest Gateway ayrı bir bileşendir (ayrı konteyner/pod, ayrı network policy). Yalnızca M2 allowlist IP + mTLS ile sınırlı inbound kabul eder. Ingest Gateway ile AI Worker arasında konteyner düzeyinde izolasyon zorunludur — ortak process veya ortak ağ namespace’i kullanılamaz.  
2) **Egress allowlist:** Worker yalnızca izinli hedeflere **outbound** bağlanabilir; internet geneli kapalıdır.  
   - Object Storage (S3-uyumlu: MinIO/Ceph vb.) → **read-only** (Worker rolü)  
   - Dispatch/Queue → **consume-only** (Worker rolü)  
   - Observability (log/metric/tracing) → **append-only**  
3) **Ağ segmentasyonu:** Web (PWA) ve public backend, Worker ağına **doğrudan erişemez**. Platform bile Worker’a “push” bağlantısı açamaz (platform→worker deny).  
4) **Kimlik & şifreleme:** Worker’ın izinli hedeflere erişimi **mTLS (karşılıklı TLS)** + servis kimliği ile yapılır; erişimler kısa ömürlü yetki (token/STS) ile sınırlanır.  
5) **Hard gate koşulları:** Worker job çalıştırmadan önce şunları doğrular:  
   - `requires_calibrated=true` ve kalibrasyon kanıtı **[KR-018 / KR-082]**  
   - dataset kanıt zinciri: manifest + hash + (varsa) signature + AV raporları (**[KR-072]**, **[KR-073]**)  
6) **Operasyonel erişim:** Yönetim erişimi yalnızca bastion/jump host + MFA + JIT (Just‑In‑Time: kısa süreli erişim) ile; yönetim portları public internete açılmaz.

### Kanonik Ağ Politikası Matrisi (Policy Matrix)
- Platform → Worker: **DENY** (TCP/HTTP yok)  
- Web/PWA → Worker: **DENY**  
- Worker → Object Storage: **ALLOW (outbound only, allowlist, read-only)**  
- Worker → Queue: **ALLOW (outbound only, consume-only)**  
- Worker → Platform Results API: **ALLOW (outbound only, mTLS)**
- Worker → Internet: **DENY (default)**
- M2 (EdgeKiosk) → Ingest Gateway: **ALLOW (inbound, allowlist IP + mTLS)**
- Ingest Gateway → AI Worker: **DENY (pod-to-pod)**

### Log / Audit Kanıtı (WORM)
- `SECURITY.DENY` (policy deny / mTLS_fail / allowlist_fail / signature_fail)  
- `JOB.REJECT` (precondition fail)  
- `DATASET.REJECTED_QUARANTINE` (hash mismatch / malware / QC fail)  
Tüm olaylar correlation_id ile zincirlenir; WORM/immutable log (silinemez) varsayılır.

### SDLC Kabul Kriterleri (Test)
- **Net test:** Platform’dan Worker subnet’e TCP denemesi → **başarısız**  
- **E2E test:** “calibrated yok” dataset ile job → Worker **reddeder**  
- **E2E test:** manifest hash mismatch → dataset **quarantine**  
- **CI gate:** contracts/examples validate, breaking-change detector → **geçmeden merge yok**

**İlgili maddeler:** [KR-066], [KR-017], [KR-018], [KR-040–043], [KR-072], [KR-073], EK-SOP-SEC, BÖLÜM 3, BÖLÜM 4

## [KR-071] Tek Yönlü Veri Akışı (One-way data flow) + Allowlist Yerleşimi

**Amaç:** Ham verinin (RAW/CALIBRATED) web/public platform katmanına çıkmasını engellemek; Worker’a inbound kapı açmadan veri akışını tek yönlü sabitlemek; allowlist’i yanlış yere koyup “güvenlik yanılsaması” üretmemek.

### Kanonik Akış (Zorunlu)
1) **EdgeKiosk → Platform/Ingress:** HTTPS 443 üzerinden, **mTLS cihaz kimliği (client cert)** ile.  
   - IP allowlist sadece **Ingress kapısında** (WAF/API Gateway/Reverse Proxy) **ikincil katman** olarak uygulanır.  
2) **Platform → Object Storage + Queue:** Platform hamı public API’de servis etmez; ham veri storage’da saklanır, iş job/queue ile orkestre edilir.  
3) **Worker → (pull) Queue/Storage:** Worker işi kuyruktan alır; dataset’i storage’dan okur; analiz eder.  
4) **Worker → Platform (sonuç):** Worker yalnızca **türev sonuç** (AnalysisResult / layer outputs) yazar; ham veri geri dönmez.  
5) **Web/PWA → Platform:** Web sadece sonuç okur; ham veri yok.

### Allowlist Yerleşimi (Gerçek Dünya Gerekçesi)
- IP allowlist **kimlik değildir**: saha şartlarında dinamik IP, CGNAT, operatör NAT, modem değişimi, farklı uplink’ler yaygındır.  
- Bu nedenle allowlist “kimlik doğrulama” yerine geçemez; **mTLS cihaz kimliği** ana kontroldür.  
- Allowlist yalnızca “giriş yüzeyini daraltma” (attack surface reduction) amaçlıdır.

### Cihaz Kimliği ve Sertifika Yönetimi (mTLS)
- EdgeKiosk cihazları kayıt altındadır: `station_id`, `device_id`, sertifika seri no, iptal/rotasyon prosedürü.  
- Sertifika iptali (revocation) ve yenileme (rotation) saha operasyon SOP’sinde zorunludur.

### SDLC Kabul Kriterleri
- Allowlist dışı IP → ingress **deny + audit**  
- Sertifikasız/yanlış sertifikalı istek → **deny + audit**  
- Worker’a doğrudan HTTP çağrısı denemesi → **deny (policy)**

**İlgili maddeler:** [KR-017], [KR-070], [KR-072], [KR-073], EK-SOP-SEC

## [KR-072] Dataset Lifecycle (Veri Yaşam Döngüsü) + Kanıt Zinciri (Chain of Custody) — Contract-First

**Amaç:** “Veri akışı sözleşmesiz” kalmasın; saha/merkez arasında tamper (kurcalama) ve malware (zararlı) tartışmaları teknik kanıtla çözülsün; Worker yalnızca kanıtlı ve kalibre edilmiş dataset’lerle çalışsın.

### Dataset = Birinci Sınıf Domain Nesnesi
Ham görüntü bir “dosya yığını” değil, sistemde **Dataset**’tir. Dataset bir durum makinesidir ve geçişler event/audit ile ispatlanır.

**Kanonik durumlar (minimum):**
- `RAW_INGESTED` (ham alındı, karantina)
- `RAW_SCANNED_EDGE_OK` (AV1 geçti)
- `RAW_HASH_SEALED` (manifest + hash mühürlendi)
- `CALIBRATED` (**[KR-018] hard gate**)
- `CALIBRATED_SCANNED_CENTER_OK` (AV2 + verification OK)
- `DISPATCHED_TO_WORKER`
- `ANALYZED`
- `DERIVED_PUBLISHED` (platforma sadece türev/sonuç)
- `ARCHIVED` (soğuk depolama)
- Hata/şüphe: `REJECTED_QUARANTINE`

### Kanıt Zinciri (Zorunlu Artefact’ler)
Her Dataset için aşağıdaki artefact’ler üretilecek ve ilişkilendirilecek:
- `dataset_manifest.json` (dosya listesi, boyut, zaman, drone_id, flight_id, mission_id, station_id)  
- `manifest.sha256` (manifest hash’i)  
- `signature.sig` (opsiyonel ama önerilen: cihaz anahtarıyla imza)  
- `scan_report_edge.json` (AV1)  
- `scan_report_center.json` (AV2)  
- `verification_report.json` (manifest’teki hash’ler ile merkezin hesapladığı hash’ler karşılaştırması)  
- `calibration_result.json` + `qc_report.json` (kalibrasyon + kalite kanıtı)

**Kurcalama ispatı (tamper proof):** Edge manifest’i ile merkezde hesaplanan hash’ler uyuşmazsa → `REJECTED_QUARANTINE` + kanıt.

### Transfer Batch (Saha Gerçeği)
Bağlantı zayıf/kesintili olabileceği için transfer **chunk/resumable** (parça/parça devam edebilir) tasarlanır. `transfer_batch` ile:
- hangi parçalar gitti/kaldi,
- hash doğrulama,
- tekrar deneme (retry),
- kesintiden devam (resume)
kanıtlanır.

### Contract-First Zorunluluğu
Bu bölüm “doküman” değil, **contracts**’ta JSON Schema’dır (bkz. `tarlaanaliz-contracts`).  
`Dataset`, `DatasetManifest`, `ScanReport`, `VerificationReport`, `CalibrationResult`, `QCReport`, `TransferBatch` şemaları kanoniktir (**[KR-081]**).

### SDLC Kabul Kriterleri
- Hash mismatch senaryosu otomatik test: “1 byte değiştir” → merkez verification **FAIL**  
- “calibrated yok” → job yaratma/çalıştırma **REJECT**  
- Contracts CI: örnek JSON’lar şemaya uymazsa **merge yok**

**İlgili maddeler:** [KR-018], [KR-070], [KR-071], [KR-073], [KR-081]

## [KR-073] Untrusted File Handling + Malware (AV1/AV2) + Güvenli Dönüştürme

**Amaç:** Ham görüntü dosyalarının (TIFF/JPEG/RAW vb.) saldırı yüzeyi olmasını engellemek; iki aşamalı tarama ve güvenli dönüştürme ile hem yereli korumak hem de “hile yapıldı mı?” sorusuna kanıt üretmek.

### Untrusted Input Prensibi
- Ham dosyalar **her zaman untrusted** kabul edilir.  
- Parse/convert işlemleri izole sandbox içinde yapılır; worker ve platform public katmanları hamı servis etmez.  
- “Güvenli türev” (tiles/thumbnails/COG) üretimi merkezde, ayrı sandbox işçisinde yapılır.

### İki Aşamalı Malware Taraması (Zorunlu)
- **AV1 (EdgeKiosk):** Yerel istasyonu korur; bariz zararlıyı erken yakalar. Çıktı: `scan_report_edge.json`.  
- **AV2 (Merkez Security Gateway / Ingestion):** İkinci kontrol + kurcalama ispatı zincirinin parçası. Çıktı: `scan_report_center.json`.  
- **Verification:** AV2 aşamasında ayrıca hash doğrulaması yapılır (`verification_report.json`) (**[KR-072]**).

### Platform Tarama Politikası (scan_policy) — İki Modlu AV2

AV2 çalışma modu `scan_policy` parametresiyle belirlenir:

**SMART (varsayılan):** Hafif doğrulama (SHA-256 + dosya tipi whitelist) sonrası tam AV2
taraması SADECE anomali tetikleyicilerde çalışır:
  - Dosya boyutu manifest'ten ±%20 sapma
  - Whitelist dışı dosya uzantısı
  - SHA-256 hash uyuşmazlığı
  - İlk kez görülen drone_id
  - Güncel olmayan EdgeKiosk yazılım sürümü
Anomali yoksa AV2 atlanır — hash + whitelist yeterli. Operasyonel maliyet düşer.

**BYPASS (istisna — KR-073'ü geçersiz kılmaz):**
AV2 tamamen devre dışı; sadece SHA-256 hash + dosya tipi whitelist.
BYPASS yalnızca **CENTRAL_ADMIN** tarafından **coğrafi (il bazında)** ve **zamansal
(tarih/saat aralığı)** olarak sınırlandırılmış biçimde etkinleştirilebilir.
Kapsam dışı iller ve zaman dilimleri SMART modda çalışmaya devam eder.
  - 72 saat maksimum TTL — süre dolunca otomatik SMART'a döner
  - Dashboard kırmızı uyarı banner'ı
  - Her etkinleştirme/uzatma ayrı audit kaydı
  - Başlangıç kapsamı: Diyarbakır + CENTRAL_ADMIN'in belirlediği iller

**İl Bazlı MANDATORY (admin override):** Belirli il kodu için global mod ne olursa
olsun tam AV2 zorunlu.

**Her modda sabit kurallar:** Hash doğrulama + dosya tipi whitelist ASLA atlanamaz.

### Quarantine Politikası
- AV fail, hash mismatch, QC fail gibi durumlarda dataset `REJECTED_QUARANTINE`’a alınır.  
- Quarantine kararları `quarantine_event` ile event/audit’e yazılır.

### SDLC Kabul Kriterleri
- (Lab) EICAR gibi test vektörü ile AV tetiklenmesi  
- “AV1 PASS ama AV2 FAIL” senaryosu → dataset quarantine + audit kanıtı  
- “parse crash” olmamalı: sandbox içinde kontrollü hata ve izolasyon
- scan_policy=SMART + anomali yok → AV atlanır, hash + whitelist geçer
- scan_policy=SMART + boyut sapması → tam AV tetiklenir
- scan_policy=BYPASS + 72 saat aşımı → otomatik SMART'a dönüş
- BYPASS etkinleştirme → audit log kaydı zorunlu
- İl bazlı MANDATORY + global BYPASS → ilgili il tam AV

**İlgili maddeler:** [KR-070], [KR-071], [KR-072]

## [KR-080] Ana İş Akışları için Teknik Kurallar

Bu bölüm; ana iş akışlarının iş planı anlatısında zaten bulunan kısımlarını tekrar etmez. Sadece teknik spesifikasyonda eklenen/sertleştirilen kuralları listeler.

**Tarla Ekleme:**
- il/ilçe/mahalle/köy + ada/parsel + alan (m²); sistem FieldID üretir
- Tekil kayıt kuralı: il+ilçe+mahalle/köy+ada+parsel kombinasyonu tekrar edemez
- Bitki türü tanımı (tarla bazında); birden fazla bitki varsa alan (m²) ile ayrı ayrı
- Analiz talebi: bitki türü bazında; yıllık abonelik veya tek seferlik
- Bitki türü değişimi: yılda 1 defa, sadece 1 Ekim - 31 Aralık (backend kuralı)

**Kooperatif:**
- Başvuru: evrak + 'Onay Bekliyor' durumunda açılır; admin onayı ile 'Aktif' olur
- Üye daveti: QR / 6 haneli davet kodu; çiftçi hesabından bağlantıyı onaylar (onay olmadan kooperatif adına işlem kısıtlanabilir)
- Tarla tekil; sahiplik/erişim ayrı yaklaşımı önerilir (çakışma/yanlış ekleme riskine karşı)

**Pilot:**
- Pilot, `config/drone_registry.yaml` ile kayıtlı drone modellerini kullanır (Faz 1: DJI Mavic 3M); drone seri numarası doğrulama referansıdır
- Günlük teslim: uçuş çıktısı her gün hafıza kartı ile yerel istasyona aktarılır
- Her uçuşta farklı hafıza kartı kullanılır. DJI tarafındaki kart kilidi/PIN gibi özellikler varsa opsiyonel ek katmandır; güvenlik garantisi değildir (bkz. [KR-015])
- Eşleştirme zorunlu: Yüklenen çekimler kayıtlı tarla ile eşleşmezse işlenmez
- Pilot sonuç raporunu görmez; sadece operasyon durumu görür

**İl Operatörü:**
- İl operatörü PII görmez; sadece ticari KPI ve kapasite planlama metrikleri görür
- Coğrafya: ilçe düzeyi veya 1-2 km grid yoğunluk katmanı (pseudonymous)

## EK-SOP-SEC: Saha Güvenlik Checklist (Pilot + İstasyon)

Bu ek, saha operasyonunda veri bütünlüğü ve zincir izlenebilirliğini (chain-of-custody) korumak için minimum kontrol listesidir.

### Pilot Checklist
- [ ] Görev dosyası (KMZ) yalnızca kanonik formatta olmalı ve MissionID/ParcelRef/Date alanları doğrulanmalı
- [ ] Uçuş sonrası SD kartta beklenmeyen dosya/klasör var mı hızlı kontrol (allowlist dışı ise istasyona bildirim)
- [ ] SD kart, mühür/etiket ile istasyona teslim; teslim anında CardID/MissionID eşleştirmesi yapılır

### İstasyon (Kiosk/Karantina) Checklist
- [ ] CardID + MissionID + BatchID kaydı açılır; giriş zamanı ve teslim alan kişi loglanır
- [ ] Dosya allowlist/MIME/size kontrolü yapılır; uygunsuz içerik karantinaya alınır
- [ ] SHA-256 manifest üretilir veya doğrulanır; hash uyuşmazlığı varsa karantina + alarm
- [ ] AV taraması (EICAR dahil test prosedürü) geçmeden hiçbir dosya işlenmez
- [ ] **Karantina PC internetsiz**; dış medya steril prosedürü uygulanır; çıktılar imzalı paket olarak aktarılır
- [ ] Upload/queue sonrası worker sonucu geldiğinde MissionID üzerinden sonuç-tarla eşleşmesi doğrulanır
- [ ] Tüm adımlar audit log'a yazılır; günlük karantina raporu üretilir

### Minimum Kanıt (Denetim İçin)
- Manifest dosyası + hash raporu
- AV tarama raporu
- Karantina kayıtları (varsa)
- MissionID tabanlı ingest/işleme log zinciri

## 3.1 Log Şeması + Correlation ID

Bu standart, **istasyon offline** akışları dahil uçtan uca iz sürmek (trace) ve denetim (audit) için **zorunludur**.
PII minimizasyonu gereği loglar **PII içermez**; `actor_id_hash` gibi tek-yönlü kimlikler kullanılır.

### Zorunlu Alanlar (JSONL — her satır 1 olay)

[EK: Kod/Şema örnekleri bu sürümden çıkarıldı. (Contracts repo’da kanonik)]

**Kurallar:**
- `correlation_id` **her zaman zorunlu** ve tüm servislerde aynen taşınır
- `error.message` max 120 karakter, **PII yok**, stacktrace ayrı (sadece güvenli ortamda)

### Correlation ID Üretimi ve Taşınması (Propagation)

- **Üretim noktası:** `station_kiosk` ingest başlatınca üretir: `corr_<16-32 hex>`
- **HTTP header:** `X-Correlation-Id: <correlation_id>` zorunlu
- **W3C Trace Context:** Mümkünse `traceparent` kullanılır; yoksa sadece correlation id yeterlidir
- **Offline paketleme:** `manifest.json` içine `correlation_id` yazılır ve paket imzasına dahil edilir

### Audit Log (WORM) — Minimum Olay Seti

WORM/immutable audit olayları:
- `PRICING.UPDATE` (admin) — eski/yeni snapshot hash + kim yaptı
- `RBAC.ROLE_ASSIGN/ROLE_REVOKE`
- `EXPERT.ASSIGN/UNASSIGN`
- `INGEST.QUARANTINE/RELEASE`
- `JOB.CANCEL`
- `SECURITY.DENY` (mTLS fail, allowlist fail, signature fail)
- `RESULT.SIGNATURE_FAIL`
